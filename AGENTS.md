# Codex Working Rules

このリポジトリで作業する Codex は、以下のルールに従うこと。

## Python Environment

- この repo では Python は `3.13` を標準とする。
- 仮想環境を新規作成する場合は、必ず `uv venv --python 3.13` を使う。
- `uv run`、pytest、依存解決で Python バージョン起因の不整合が疑われる場合は、まず Python `3.13` 環境での再現可否を優先確認する。
- README やセットアップ例に仮想環境作成手順を書く場合も、`3.13` を明記する。

## Existing Pattern First

- 実装前に既存コードを確認し、ファイル名、CLI 入口、テスト配置、出力形式を既存パターンに揃える。
- `flows/` 配下の新規フロー関連ファイルは、既存の命名に合わせて `*_flow.py` とする。
- 新規テストは、対象に対応する `tests/test_*.py` として追加する。
- README の使用例やコマンド例は、追加した入口に合わせて更新する。

## Spec Lock Before Edit

- ユーザーが指定した命名、出力項目、公開インターフェースは、編集前に固定し、勝手に拡張しない。
- stdout や戻り値に出す項目は、要求された最小集合だけに絞る。
- 追加項目が必要だと判断した場合は、実装前に根拠を明示する。明示要求がない限り増やさない。

## Implementation Discipline

- 既存の機能やフローを流用し、重複実装を避ける。
- 既存フロー本体を変更しなくてよい場合は、入口側だけを追加する。
- 関係ないファイルや既存の dirty changes を巻き戻さない。
- 変更範囲は、要求を満たす最小限に保つ。

## Verification Discipline

- 追加・変更した箇所に直接対応するテストを最低限実行する。
- 全件テストが環境制約で無理な場合は、関連テストを限定実行し、失敗理由をコード要因か環境要因か切り分けて報告する。
- Python バージョン差異が失敗原因なら、その点を明示し、Python `3.13` 前提の確認手順を書く。
- 実行していないテストを、実行済みとして扱わない。

## Final Response Contract

- 最終報告では、変更ファイル、実行した検証、未検証事項、残る制約を短く明示する。
- 実際に確認した範囲だけを書く。
- 環境制約で確認できなかったことがあれば、省略せず記載する。

## Repo-Specific Examples

- フロー追加時は `flows/*_flow.py` に揃える。
- CLI 出力は、要求されたキーのみを出力する。
- 新規コードには対応テストを追加する。
- 仮想環境作成は `uv venv --python 3.13` を使う。
