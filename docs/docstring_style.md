# Docstring Style Guide

このプロジェクトでは、関数・メソッド・クラスの docstring を日本語で統一し、実装意図と I/O 契約をコード上で追える状態を維持します。Codex を含む自動編集でも、このルールをデフォルトとして扱います。

## 基本方針

- Docstring は三重ダブルクォートで記述する。
- 先頭 1 行で、その対象の目的を簡潔に言い切る。
- 先頭行の後に空行を 1 行入れ、以降は固定セクションで記述する。
- セクション名は `目的` `処理内容` `入力` `出力` `例外` を必須とする。
- 外部システムやファイル、ネットワーク、DB、Prefect Block などに依存する場合は `外部依存リソース` を追加する。
- 説明は実装詳細の羅列ではなく、呼び出し側が知るべき契約と副作用を中心に書く。
- 型ヒントがあっても `入力` と `出力` で意味を補足する。
- 例外は「何が起きたら、どの例外を送出するか」を具体的に書く。
- 実装と docstring がずれたら、コード変更と同じコミットで docstring も更新する。

## 推奨テンプレート

```python
def example(value: str) -> dict[str, str]:
    """入力値を正規化して設定辞書へ変換する。

    目的:
        呼び出し側が扱いやすい共通形式へ値をそろえる。
    処理内容:
        文字列を検証し、必要な正規化を行って辞書へ格納する。
    入力:
        value: 変換対象の入力文字列。
    出力:
        dict[str, str]: 正規化済み値を含む辞書。
    例外:
        ValueError: 入力文字列が空または不正な形式の場合。
    外部依存リソース:
        なし。
    """
```

## 対象ごとの運用

- 関数:
  - 必須セクションをすべて記載する。
  - 副作用がある場合は `処理内容` に明記する。
- クラス:
  - クラスの責務と保持する状態の意味を `目的` と `処理内容` に記載する。
  - `出力` は通常 `なし。` とし、必要なら生成インスタンスの役割を書く。
- `__init__`:
  - 初期化対象の状態、入力引数、想定する前提条件を明記する。
- Prefect の `@task` / `@flow`:
  - 外部依存リソースを必ず書く。
  - リトライ、外部 API、永続化などの副作用を `処理内容` に含める。

## 記載ルール

- `入力` は引数ごとに `name: 説明。` の形式で列挙する。
- `出力` は型名を先頭に書き、その値に含まれる重要項目を説明する。
- `例外` は送出しない場合でも `なし。` と書く。
- `処理内容` は 1 文で終わらせず、主要な分岐や段階が分かる粒度で書く。
- `目的` は「なぜこの処理が存在するか」を書き、単なる処理の言い換えにしない。
- 実装上の一時的な事情や TODO を docstring に書かない。

## Codex 向け運用ルール

- Codex が新規関数・メソッド・クラスを追加する場合は、このガイドのテンプレートに従って docstring を作成する。
- Codex が既存実装を変更し、目的・入出力・例外・外部依存のいずれかが変わる場合は、同じ編集で docstring も更新する。
- Docstring が簡略形で残っている既存コードを編集対象に含めた場合は、周辺の書式に合わせて構造化 docstring へそろえる。
